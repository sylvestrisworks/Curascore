## Issues Found

### LOW — API key in RAWG URL query parameter

**File:** `src/app/api/cron/sync-game-updates/route.ts` ~line 48

**Description:** The RAWG API key is appended as a URL query parameter:

```
const url = `https://api.rawg.io/api/games?key=${RAWG_API_KEY}&ordering=...`
```

Query-parameter API keys are logged by most HTTP infrastructure (Vercel request logs, APM tools, error trackers like Sentry). If any future `console.log(url)` or an unhandled error surfaces the URL, the key is exposed in plaintext.

The key is not currently printed to logs, but the risk is latent.

**Recommendation:** Abstract RAWG calls into a helper that passes the key via a header (if RAWG supports it) or at minimum ensure the URL is never passed to `console.error`/`console.log`. If RAWG requires query-param auth, document this constraint and confirm the key is excluded from Vercel log shipping.

### LOW — Full error string returned in cron API response

**File:** `src/app/api/cron/review-games/route.ts` ~line 318 (outer catch block)

**Description:** The fatal-error catch block returns the raw JS error string to the caller:

```
return NextResponse.json({ error: String(err) }, { status: 500 })
```

Although this route is protected by `Authorization: Bearer <CRON_SECRET>`, leaking internal error details (DB error messages, service URLs, stack fragments) in HTTP responses is bad practice. It unnecessarily surfaces implementation details to whoever holds the cron token.

**Recommendation:** Log the full error server-side and return a generic message:

```tsx
console.error('[review-games] Fatal error:', err)
return NextResponse.json({ error: 'Review job failed' }, { status: 500 })
```
