## Findings

### LOW — Inconsistent `userId` extraction in child-profiles routes

**File:** `src/app/api/child-profiles/[id]/route.ts`

**Lines:** `getUserId()` helper

**Description:** The `GET` / `POST` handler in `child-profiles/route.ts` extracts the user identifier exclusively as `session?.user?.id`. The `PUT` / `DELETE` handler in `[id]/route.ts` uses a `getUserId()` helper that falls back to `session?.user?.email` when `id` is absent. If a user authenticates via a provider that populates `email` but not `id` in the NextAuth session, their profile records could be written with one identifier (email) and queried with another (id), or vice-versa. In an extreme case where two different OAuth providers share an email address, this could theoretically allow cross-account access to child-profile mutations.

**Recommendation:** Standardise on a single identifier. Extract userId the same way in both files — prefer `session.user.id` and reject the request if it is absent, rather than silently falling back to email.

### LOW — Gemini API key exposed as URL query parameter

**File:** `src/app/api/cron/debate-games/route.ts`

**Line:** `const url = \`${GEMINI_URL}?key=${process.env.GEMINI_API_KEY}``

**Description:** The Gemini API key is appended as a `?key=` query parameter. This is the standard Gemini REST interface and is not a code defect, but query parameters are frequently captured in server access logs, CDN logs, and error-reporting tools (e.g. Sentry breadcrumbs). If any of those log sinks are less strictly controlled than the secret store, the key could leak.

**Recommendation:** Consider switching to the `x-goog-api-key` request header instead, which keeps the secret out of URLs and therefore out of most access logs. This is a supported Gemini authentication alternative.

### LOW (pre-existing) — `unsafe-inline` in `script-src` CSP directive

**File:** `next.config.js`

**Description:** The Content Security Policy includes `script-src 'self' 'unsafe-inline'`. This weakens XSS protection because any inline script — including one injected via a reflected or stored XSS vulnerability — would be permitted. The comment acknowledges this is required for `next-intl` hydration and JSON-LD tags. This issue is **pre-existing** and was not introduced by today's changes (the commit only added image CDN domains).

**Recommendation:** Investigate replacing `'unsafe-inline'` with a nonce-based or hash-based CSP using Next.js middleware, which is supported from Next.js 13.4+. This is a medium-effort improvement with meaningful XSS hardening benefit.

### INFO — Minor inconsistency in `Authorization` header casing

**Files:** `src/app/api/cron/debate-games/route.ts` vs `translate-content/route.ts`

**Description:** `debate-games` uses `req.headers.get('authorization')` (lowercase); `translate-content` uses `req.headers.get('Authorization')` (title case). Next.js normalises incoming HTTP headers to lowercase, so both work correctly in practice. No security risk, but the inconsistency is worth cleaning up for readability.
