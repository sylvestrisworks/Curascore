## Issues Found

### 🟡 MEDIUM — Unbounded `gameIds` array in Steam import (confirm action)

**File:** `src/app/api/import/steam/route.ts`

**Description:** The Zod schema for the `confirm` action accepts `z.array(z.number().int().positive())` with no `.max()` cap. An authenticated user could POST `{"action":"confirm", "gameIds": [1,2,...,50000]}`, triggering a large DB ownership check (`SELECT` over `userGames`) followed by a large batch `INSERT`. Rate limiting (5 req/min) is present but does not bound the cost of a single request.

**Recommendation:** Add `.max(500)` (matching the Steam preview limit of 200 games plus headroom) to the `gameIds` array:

```tsx
z.object({ action: z.literal('confirm'), gameIds: z.array(z.number().int().positive()).max(500) })
```

### 🟡 MEDIUM — `unsafe-inline` in `script-src` CSP

**File:** `next.config.js`

**Description:** The Content-Security-Policy for all non-studio routes includes `script-src 'self' 'unsafe-inline'`. This means any XSS injection that achieves inline script execution is not blocked by the CSP. The comment in the file correctly notes this is required for next-intl hydration and JSON-LD tags — it is a known Next.js App Router constraint, not a new regression.

**Recommendation:** Track as tech debt. When feasible, migrate to nonce-based CSP (Next.js 15+ supports this via middleware). In the interim, the existing `X-Frame-Options: DENY` and React's default output escaping provide meaningful layered defence. No immediate action required, but should not be widened further.

### 🔵 LOW — `/studio` path entirely exempt from CSP

**File:** `next.config.js`

**Description:** The Sanity Studio path (`/studio(.*)`) has no `Content-Security-Policy` header, only `X-Content-Type-Options` and `Referrer-Policy`. This is intentional to support Sanity's varied API calls, but it means any XSS within the studio context is fully unprotected. Risk is reduced because studio access should be limited to authenticated admins.

**Recommendation:** Verify that `/studio` is not publicly accessible (e.g., Vercel password protection, or Sanity's own auth gate). If admin access is confirmed restricted, this is acceptable. If the studio is reachable by anyone with a Sanity token, consider adding a minimal allowlist CSP targeting known Sanity hostnames.

---

### 🔵 LOW — Full error string returned in cron route 500 response

**File:** `src/app/api/cron/ingest-games/route.ts`

**Description:** The outer catch block returns `{ error: String(err) }`, which may expose internal error messages (file paths, DB connection strings in stack traces) to the caller. Risk is low because the route requires `Authorization: Bearer <CRON_SECRET>` and is only called by GitHub Actions.

**Recommendation:** Low priority. Replace `String(err)` with a generic message and ensure the full error is only emitted to server-side logs:

```tsx
console.error('[ingest] Fatal error:', err)
return NextResponse.json({ error: 'Internal error — check server logs' }, { status: 500 })
```
