## Issues Found

### BUG

**`src/app/api/cron/debate-games/route.ts` — Unvalidated AI response written to database (persistent from 2026-04-20 review)**

The Azure tool call response is parsed with `as any` and no schema validation:

```tsx
// eslint-disable-next-line @typescript-eslint/no-explicit-any
const a = JSON.parse(toolCall.function.arguments) as any
return {
  scores: { b1: a.b1, b2: a.b2, b3: a.b3, r1: a.r1, r2: a.r2, r3: a.r3 },
  reasoning: a.reasoning,
}
```

This was flagged in the 2026-04-20 code review and is still unresolved after the Gemini → Azure migration. If Azure returns a malformed or partial tool response (field missing, wrong type, extra nesting), corrupt scores land silently in `game_scores`. The fix is a Zod schema for each score group, applied before passing to `weightedDebateScores`:

```tsx
const scoreGroupSchema = (fields: string[]) =>
  z.object(Object.fromEntries(fields.map(f => [f, z.number().int().min(0)])))

const debateScoresSchema = z.object({
  b1: scoreGroupSchema(B1_FIELDS), b2: scoreGroupSchema(B2_FIELDS), b3: scoreGroupSchema(B3_FIELDS),
  r1: scoreGroupSchema(R1_FIELDS), r2: scoreGroupSchema(R2_FIELDS), r3: scoreGroupSchema(R3_FIELDS),
  reasoning: z.string(),
})
```

This is the second consecutive review this appears in — it should be the top priority before the next cron run.

### IMPROVEMENT

**`src/app/api/cron/review-games/route.ts` — Type narrowing gap on partial SELECT result**

The commit correctly scopes the SELECT to 19 columns, but `buildReviewPrompt` and `saveReview` still declare `game: GameRow` (`typeof games.$inferSelect` — the full table type). The actual value passed is a Drizzle partial-select row, which is a subset type. TypeScript strict mode should flag this as non-assignable.

If a future developer adds a field access to `buildReviewPrompt` for a column *not* in the scoped SELECT (e.g. `g.coverUrl`), they'll get `undefined` at runtime with no compile-time warning. Fix: declare a `ReviewGameRow` type matching only the 19 selected fields, or use `typeof pending[number]['games']` as the parameter type.

### NITPICK

**`scripts/translate-messages.ts` — No env guard before using `AWS_BEARER_TOKEN_BEDROCK`**

The Bedrock call uses `process.env.AWS_BEARER_TOKEN_BEDROCK` directly in the `Authorization` header. If the env var is unset in a local run, the request sends `Bearer undefined` and Bedrock responds with a cryptic 401/403. The `review-games` route has an explicit guard (`if (!process.env.AWS_BEARER_TOKEN_BEDROCK) return error`) — `translate-messages.ts` should add a similar check at the top of `main()` to fail fast with a clear message.

---

**`.env.example` — `AZURE_OPENAI_ENDPOINT` looks like a real internal URL**

The example value `https://lumikin-openai.openai.azure.com` uses the actual project name. Convention in this file (and industry practice) is to use generic placeholders. Should be:

```
AZURE_OPENAI_ENDPOINT=https://your-resource-name.openai.azure.com
```

Low-stakes since this is `.env.example`, not `.env`, but worth consistency.

The 2026-04-20 review called out `debate-games/route.ts` for writing unvalidated Gemini responses to the database. The provider changed (Gemini → Azure OpenAI) but the validation gap did not close. This pattern risks silent data corruption in the borderline-score pipeline — the one area where accuracy matters most for the curascore. A one-time investment in a shared Zod schema for debate scores (`src/lib/scoring/debateSchema.ts`) would fix this permanently and be reusable in tests.
